Document Overview & Description

This document provides the configuration and customization settings that have been implemented. This configuration and customization will serve as the baseline for a full release.

  • Only sections that have modifications over Salesforce out-of-the-box (OOB) settings are documented.
  • This document is laid out in a consistent manner with the Salesforce customize and configuration settings accessed from Setup | Customize in the Salesforce application.
  • This document does not detail standard Salesforce functionality unless such functionality is requested via a requirement.

The document will provide an in-depth technical understanding of the customization and configuration made to the Salesforce application. This section is intended to provide guidance to future developers by providing an understanding of the modifications that were made to the Salesforce objects, fields, profiles, roles, etc. In addition, this section includes an understanding of how to execute the batch upload, with screen shots and troubleshooting tips for common errors that are encountered.
There are five main sections in the document:

  • High-Level Data Model – defines the underlying data structure for CRM Implementation application.
  • Custom Functionality Detail Design – Provides the details of some custom components/functionalities that have been developed.
  • Customize – defines the application customizations in terms of labels, workflows, user management, apex, triggers, VisualForce components, reports and dashboards. This section will be defined in Phase
  • Security – defines the PCI compliance security configurations, Salesforce security settings and details of PCI Audit log configurations.

Integrations

A typical enterprise uses many applications, many or most of which are not designed to work with one another out of the box. Integrating separate but related apps helps organizations achieve greater levels of operational consistency, efficiency, and quality. This section describes the architecture, data flow and components for integration related components build in the system.

Security Integration:
Integrate authentication mechanisms across applications to improve the user experience and minimize user administration.

UI Integration:
Combine the UIs of two or more apps to create composite apps with little or no rework on the UI of each individual app.

Logic Integration:
Business logic often spans multiple application systems. Extending related business logic from one app to another helps to implement complete end to end business processes.

Data Integration:
Integrating applications at the data layer is a common scenario. For example, multiple apps written in different programming languages can all use an open API and manage related data in one shared database.

{{#if ssoSettings}}

Single Sign-On

Single Sign-On (SSO) is an authentication process that allows users to access multiple applications such as Salesforce with a single set of credentials. This is done by creating a single identity that can be used across multiple applications and websites. In Salesforce, SSO allows users to access Salesforce with credentials from other identity providers such as ADFS, Google, Microsoft, or any SAML-based identity provider. This simplifies the user experience, as they don't need to remember multiple sets of credentials.
